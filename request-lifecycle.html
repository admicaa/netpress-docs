Core Ideas

Request Lifecycle

Netpress keeps the request path intentionally short: boot providers, match a route, run middleware, execute the controller or terminal handler, then let the smart-response layer turn the return value into an HTTP response.

Quick Example

// bootstrap/app.js
import express from "express";
import providers from "./providers.js";
import Handler from "../app/Exceptions/Handler.js";

export async function createApp() {
  const app = express();

  for (const Provider of providers) {
    await Provider.register(app);
  }

  app.use(Handler.handle);
  return app;
}

Lifecycle Order

  1. server.js loads .env, calls createApp(), and starts listening.
  2. bootstrap/app.js creates the Express app.
  3. bootstrap/providers.js boots providers in a fixed order.
  4. AppServiceProvider installs global middleware, registers the view renderer, and decorates res with helpers like success, error, paginate, and component-aware render.
  5. DatabaseServiceProvider boots the configured database runtime and shared models.
  6. AuthServiceProvider, MailServiceProvider, and EventServiceProvider register their integrations.
  7. RouteServiceProvider mounts:
  • routes/web.js at / only when config/view.js enables a web engine
  • routes/api.js under /api/v1
  1. Route middleware runs.
  2. The terminal handler returns a value.
  3. Netpress dispatches that return value as JSON, HTML, redirect, text, raw data, or 204 No Content.
  4. Thrown errors flow to the global exception handler.

Smart Response Dispatch

Routes created with Router() automatically wrap the last non-error handler in the smart-response dispatcher.

That means a handler can return:

  • a plain object or array and get JSON automatically
  • view(...)
  • json(...)
  • redirect(...)
  • a string and get text/plain
  • null or undefined and get 204 No Content

Manual res.send() and res.json() still work. The dispatcher stands down as soon as the response has already been handled.

View Flash State

When a controller returns:

return redirect("/form")
  .withErrors({ email: ["Required"] })
  .withInput(req.body);

AppServiceProvider and shareViewState make that flash data available to the next server-rendered view as:

  • errors, an ErrorBag
  • old(field, fallback)

The flash data is consumed on the next request and then cleared.

Why Provider Order Matters

The default provider order is deliberate:

  • views and response helpers are ready before routes mount
  • database services are available before auth or model-backed services need them
  • routes mount after the core services are already registered

If you add a custom provider, place it according to what it depends on and what depends on it.
