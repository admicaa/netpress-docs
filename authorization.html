Security

Authorization

Authorization decides what an authenticated user may do. NetPress uses a small policy pattern plus role middleware, which feels close to Laravel without adding extra ceremony.

Quick Example

// app/Policies/UserPolicy.js
export default class UserPolicy {
  async update(authUser, targetUser) {
    return authUser.role === "admin" || authUser.id === targetUser.id;
  }
}

Use it in a controller:

const targetUser = await UserService.findById(req.params.id);
await this.authorize(req, UserPolicy, "update", targetUser);

Explanation

There are three common authorization paths:

  • requireRole(&quot;admin&quot;) for simple role checks
  • authorize(PolicyClass, method, resourceFn) for route middleware checks
  • this.authorize(req, PolicyClass, method, resource) inside a controller action

Policies receive the authenticated user and the target resource. They return true or false.

Additional Examples

Role-only check:

router.group(authMiddleware, (router) => {
  router.group(requireRole("admin"), (router) => {
    router.get("/users", UserController.index);
  });
});

Route middleware policy check:

router.delete(
  "/users/:id",
  authMiddleware,
  authorize(UserPolicy, "delete", async (req) => {
    return UserService.findById(req.params.id);
  }),
  UserController.destroy,
);

Post policy example:

export default class PostPolicy {
  async update(authUser, post) {
    return authUser.role === "admin" || authUser.id === post.userId;
  }
}

Advanced Usage

Keep policies close to the domain language of the app. When rules become more complex, call a service from the policy instead of turning route files into authorization logic.

Notes / Tips

  • Use roles for broad access.
  • Use policies for record-level access.
  • Return false from the policy and let the helper raise the 403 response.
