Core Ideas

Service Providers

Service providers are the boot sequence of the starter. They register container bindings, mount middleware, wire routes, and expose optional integrations — all in a predictable order.

Netpress uses a hybrid bootstrap: providers have a cheap register() phase that always runs, and a boot() phase that can be deferred. See Hybrid Bootstrap for the full model.

Provider Shape

import { ServiceProvider } from "@admicaa/netpress";

export default class MyServiceProvider extends ServiceProvider {
  static deferred = false;           // true = skip boot() at startup
  static provides = [];              // container keys this provider owns

  async register() {
    // cheap bindings — runs for every provider at boot
  }

  async boot() {
    // heavy startup — runs eagerly, or on first `app.make(key)` if deferred
  }
}

this.app is the Netpress Application kernel. this.app.express is the underlying Express instance. this.app.container is the shared service container.

Quick Example

// bootstrap/providers.js
import AppServiceProvider from "../app/Providers/AppServiceProvider.js";
import DatabaseServiceProvider from "../app/Providers/DatabaseServiceProvider.js";
import AuthServiceProvider from "../app/Providers/AuthServiceProvider.js";
import MailServiceProvider from "../app/Providers/MailServiceProvider.js";
import EventServiceProvider from "../app/Providers/EventServiceProvider.js";
import RouteServiceProvider from "../app/Providers/RouteServiceProvider.js";

export default [
  AppServiceProvider,
  DatabaseServiceProvider,
  AuthServiceProvider,
  MailServiceProvider,         // deferred
  EventServiceProvider,
  RouteServiceProvider,
];

What The Default Providers Do

ProviderModeResponsibility
AppServiceProvidereagerglobal Express middleware, shared view renderer, cache/storage facade configuration
DatabaseServiceProvidereagerregisters the DB manager; connections stay lazy unless listed in preload
AuthServiceProvidereagermounts passport.initialize(); strategies are dynamic-imported
MailServiceProviderdeferredbinds a mail factory — nothing runs until Mail.send()
QueueServiceProviderdeferredbinds a queue factory — BullMQ is built on first dispatch or queue:work
EventServiceProvidereagerregisters event listeners and model observers
RouteServiceProvidereagermounts web and API route trees

Lazy Service Registration and Resolution

A deferred provider binds factories in register(). The factory runs the first time something pulls the key from the container:

import { ServiceProvider, container } from "@admicaa/netpress";
import Stripe from "stripe";

export default class PaymentServiceProvider extends ServiceProvider {
  static deferred = true;
  static provides = ["payments", "payments.stripe"];

  async register() {
    this.app.container.bind("payments.stripe", () => {
      return new Stripe(process.env.STRIPE_SECRET_KEY);
    });
    this.app.container.bind("payments", () => this.app.container.get("payments.stripe"));
  }
}

Resolution is uniform:

// Anywhere that holds the Application:
const stripe = await app.make("payments");

// Or from the static container for ad-hoc use:
const stripe = container.get("payments.stripe");

Neither Stripe nor its SDK is loaded until the first resolution.

A Real Route Provider Example

RouteServiceProvider mounts web routes only when a view engine is configured:

import { ServiceProvider, Router, config } from "@admicaa/netpress";
import apiRoutes from "../../routes/api.js";

export default class RouteServiceProvider extends ServiceProvider {
  async register() {
    await config.load("view");
  }

  async boot() {
    const router = Router();

    if (config.get("view.engine")) {
      const { default: webRoutes } = await import("../../routes/web.js");
      router.use("/", webRoutes);
    }

    router.group("/api/v1", (api) => {
      api.use("/", apiRoutes);
    });

    this.app.express.use(router);
  }
}

Guidelines

  • Keep register() cheap — binding factories, loading config. Save side effects for boot().
  • Use static deferred = true when the service requires a socket, subprocess, or filesystem handle.
  • Register dependencies before the providers that need them (DB before Auth, etc.).
  • Mount routes late: RouteServiceProvider should be last in the eager list.
  • To warm a deferred provider at boot, call await app.make(key) from inside an eager provider's boot().
